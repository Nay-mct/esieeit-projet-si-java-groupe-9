5. Authentification et JWT

Du login jusqu'a la route protegee

Cycle de login

  1. `/api/auth/login` recoit `login + password`
  2. `AuthenticationManager` verifie les identifiants
  3. `CustomUserDetailsService` recharge le user
  4. `PasswordEncoder` compare le hash BCrypt
  5. `JwtService` genere le token
  6. le token est renvoye au client

Cycle d'une route protegee

  1. header `Authorization: Bearer ...`
  2. `JwtAuthenticationFilter` extrait le token
  3. le token est valide
  4. le `SecurityContext` est rempli
  5. `@PreAuthorize` controle le role
  6. le controller peut s'executer

8. Qualite et bilan

Pourquoi le projet est deja presentable

Validation

Les DTO utilisent `@Valid` pour filtrer les requetes invalides.

Erreurs JSON

`GlobalExceptionHandler` gere les `400`, `401`, `403`, `404`, `409`, `500`.

Tests

Des tests metier existent sur les validations utilisateur et le workflow des taches.

Point d'attention

La creation de tache utilise encore un `defaultProject`, ce qui est acceptable pour le TP mais a raffiner ensuite.

Conclusion

Le projet n'est pas juste une suite d'endpoints

C'est un backend complet, avec architecture, logique metier, securite, persistance et un vrai parcours de demo.
